# shadcn-solid-components Documentation Conventions

This document defines the conventions for writing and maintaining documentation in the shadcn-solid-components project. Follow these rules to ensure consistency, accuracy, and a good reader experience across all doc pages.

---

## 1. Documentation Architecture

```text
docs/
├── src/
│   ├── content/
│   │   ├── index.ts              # Autogenerated — DO NOT EDIT
│   │   └── docs/
│   │       ├── components/       # Primitive component docs (one .mdx per component)
│   │       ├── hoc/              # HOC docs (one .mdx per HOC)
│   │       └── lib/              # Library/utility docs (one .mdx per module)
│   ├── examples/                 # Live example components (one .tsx per demo)
│   ├── components/
│   │   └── mdx.tsx               # MDX rendering components (ComponentPreview, etc.)
│   ├── App.tsx                   # Docs app shell & routing
│   ├── index.tsx                 # Entry point
│   └── styles.css                # Docs-only styles
├── plugins/
│   ├── content.ts                # Vite plugin: scans .mdx files, generates index.ts
│   └── mdx.ts                   # Vite plugin: MDX compilation config
├── index.html
├── package.json
├── pnpm-lock.yaml
└── tsconfig.json
```

### Rules

- **`docs/src/content/docs/`** — The only location for documentation `.mdx` files. Each file maps to one route under `/docs/<section>/<name>`.
- **`docs/src/examples/`** — Live demo components imported by `.mdx` files. One file per distinct demo variant.
- **`docs/src/content/index.ts`** — Fully autogenerated by `docs/plugins/content.ts`. **Never edit manually.** It regenerates on every Vite dev server start and on any `.mdx` file change.
- **`docs/plugins/`** — Build plugins only. Do not add documentation content here.

## 2. MDX File Structure

Every documentation page must follow this exact structure:

```mdx
---
title: ComponentName
description: One-sentence summary of what the component does.
section: Components | HOC | Libraries
toc: true
order: <optional integer for nav sorting>
status: <optional string, e.g. "beta", "deprecated">
---

import DemoName from "../../../examples/demo-name"

<Opening paragraph — what the component is and when to use it.>

## Usage

```tsx
import { ComponentName } from "shadcn-solid-components/<category>/component-name"
```

<Brief prose on how to use the component.>

## Exports

- `ComponentName`, `SubComponent` — components
- Types: `ComponentNameProps`, `SubComponentProps`

## Examples

### Default

<ComponentPreview name="Component Name Demo">
  <div class="mb-4 w-full">
    <DemoName />
  </div>

  ```tsx
  <full runnable code for the demo>
  ```

</ComponentPreview>

### Variant Name

<Repeat ComponentPreview pattern for each variant/feature demo>

## API

<Only for HOCs and complex components — list props/options in a structured format>
```

### Section Order

Sections must appear in this order. Omit sections that do not apply, but never reorder them:

1. **Frontmatter** — required
2. **Imports** — required when referencing example components
3. **Opening paragraph** — required
4. **Usage** — required (import statement + brief prose)
5. **Exports** — required (list all exported symbols and types)
6. **Examples** — required (at least one `ComponentPreview`)
7. **API** — optional, for complex prop tables or imperative APIs
8. **Installation** — optional, only when the component requires a peer dependency

## 3. Frontmatter Specification

Every `.mdx` file must begin with YAML frontmatter:

```yaml
---
title: Button
description: Clickable control for actions and navigation.
section: Components
toc: true
---
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `title` | `string` | Yes | Display title in the page and navigation. PascalCase for components. |
| `description` | `string` | Yes | One-sentence summary. Shown in navigation tooltips and SEO. |
| `section` | `string` | Yes | Must be `"Components"`, `"HOC"`, or `"Libraries"`. Determines nav grouping. |
| `toc` | `boolean` | Yes | Whether to show the table of contents sidebar. Almost always `true`. |
| `order` | `number` | No | Integer for sorting within a section. Lower values appear first. |
| `navTitle` | `string` | No | Override title for navigation only (e.g. shorter form). |
| `status` | `string` | No | Badge label such as `"beta"` or `"deprecated"`. Displayed next to the nav item. |

### Section Mapping

The `section` value determines both the navigation group and the subdirectory:

| `section` value | Directory | Nav group |
| --- | --- | --- |
| `"Components"` | `docs/src/content/docs/components/` | Components |
| `"HOC"` | `docs/src/content/docs/hoc/` | HOC |
| `"Libraries"` | `docs/src/content/docs/lib/` | Libraries |

## 4. Writing Style

### 4.1 Language

All documentation must be written in **English**.

### 4.2 Tone

- Use **present tense** and **imperative mood** for instructions: "Import the component", not "You should import the component".
- Use **second person** sparingly; prefer direct statements: "Call `confirm()` to open the dialog", not "You can call `confirm()` to open the dialog".
- Be **concise**. Avoid filler phrases like "simply", "just", "basically", "in order to".
- Avoid **anthropomorphism**: write "the component renders", not "the component wants to render".

### 4.3 Terminology

- **Component** — a primitive in `src/components/`
- **HOC** — a higher-order/composed component in `src/hoc/`
- **Library** / **Utility** — a module in `src/lib/`
- **Variant** — a visual style option defined via `cva()`
- **Sub-component** — a named part of a compound component (e.g. `DialogHeader`, `DialogTitle`)
- **Primitive** — the underlying headless component from Kobalte, Ark UI, or corvu

### 4.4 Capitalization

- **PascalCase** for component names: `Button`, `AlertDialog`, `DataTable`
- **camelCase** for function/variable names in prose: `buttonVariants`, `useComponentClass`
- **kebab-case** for import paths: `shadcn-solid-components/components/alert-dialog`
- **Sentence case** for headings: "Custom styling", not "Custom Styling"
- **Lowercase** for code keywords in prose: `as` prop, `class` prop, `variant` prop

## 5. Code Examples

### 5.1 Import Paths

All import paths in documentation must use the **public package alias**, never relative paths:

```tsx
// ✅ Correct
import { Button } from "shadcn-solid-components/components/button"

// ❌ Wrong
import { Button } from "../../src/components/button"
```

### 5.2 Completeness

Every code example must be **runnable** when copied into a SolidJS project that has `shadcn-solid-components` installed. This means:

- Include all necessary imports at the top of the code block
- Include the `export default` or `export function` declaration
- Do not omit props or children with comments like `/* ... */` unless the omitted parts are clearly unrelated to the example's purpose
- If the example uses signals or state, include the `createSignal` / `createMemo` import

### 5.3 Code Block Language

Always specify the language for fenced code blocks:

````mdx
```tsx
// component code
```

```bash
pnpm add @tanstack/solid-table
```
````

### 5.4 Inline Code

Use backticks for:

- Component names: `Button`, `DialogHeader`
- Prop names: `variant`, `size`, `as`
- Function names: `confirm()`, `useComponentClass()`
- Import paths: `shadcn-solid-components/components/button`
- Type names: `ButtonProps`, `ColumnDef<T>`

Do **not** use backticks for:

- Plain English words
- Conceptual terms (write "variants" not `` `variants` `` unless referring to a specific export)

## 6. ComponentPreview

### 6.1 Usage Pattern

Live demos must use the `<ComponentPreview>` wrapper, which renders both the live component and the source code:

```mdx
<ComponentPreview name="Button Default">
  <div class="mb-4">
    <Button>Button</Button>
  </div>

  ```tsx
  import { Button } from "shadcn-solid-components/components/button"

  export function ButtonDemo() {
    return <Button>Button</Button>
  }
  ```

</ComponentPreview>
```

### 6.2 Naming Convention

The `name` prop must follow the format: **`<Component> <Variant>`** (title case):

- `"Button Default"` — not `"button default"` or `"Default Button"`
- `"Alert Destructive"` — not `"destructive alert"`
- `"Slider Custom Label Demo"` — include "Demo" only when disambiguation is needed
- `"Data Table Demo"` — compound names use spaces

### 6.3 Layout Wrapper

Wrap the live component in a `<div>` with appropriate spacing and width:

- Default: `<div class="mb-4">`
- Full-width components (tables, sidebars): `<div class="mb-4 w-full">`
- Constrained-width components (alerts, cards): `<div class="mb-4 w-full max-w-xl">`

### 6.4 Example File vs Inline

- **Simple examples** (single component, no state) may be written inline inside the `<ComponentPreview>`.
- **Complex examples** (signals, callbacks, multiple components) must be extracted to a separate file in `docs/src/examples/` and imported:

```mdx
import DataTableDemo from "../../../examples/data-table-demo"

<ComponentPreview name="Data Table Demo">
  <div class="mb-4 w-full">
    <DataTableDemo />
  </div>

  ```tsx
  import DataTableDemo from "../../../examples/data-table-demo"
  ```

</ComponentPreview>
```

When the example is too large to show inline, the code block may reference the example file with just the import line, as shown above.

## 7. Example File Conventions

### 7.1 Location and Naming

Example files live in `docs/src/examples/`. Naming convention:

```text
<component>-<variant>-demo.tsx
```

Examples:

- `button-demo.tsx` — default variant
- `button-destructive-demo.tsx` — destructive variant
- `slider-custom-label-demo.tsx` — custom label feature
- `data-table-demo.tsx` — full-featured demo

### 7.2 File Structure

Every example file must:

1. Import from `shadcn-solid-components/...` (public alias), never from relative source paths
2. Export a named component matching the file name in PascalCase, or export as default
3. Be self-contained — no external state or context providers unless the demo specifically demonstrates them

```tsx
// button-loading-demo.tsx
import { Button } from "shadcn-solid-components/components/button"

const ButtonLoadingDemo = () => {
  return (
    <Button disabled>
      <svg ...>...</svg>
      Please wait
    </Button>
  )
}

export default ButtonLoadingDemo
```

### 7.3 One Concept Per File

Each example file should demonstrate **one variant or feature**. Do not combine multiple unrelated demos in a single file. If a component has five variants, create five separate example files.

## 8. Exports Section

Every doc page must include an **Exports** section listing all public symbols. Use two groups:

```mdx
## Exports

- `Button`, `buttonVariants` — components and utilities
- Types: `ButtonProps`
```

For compound components with many sub-components:

```mdx
## Exports

- `Dialog`, `DialogPortal`, `DialogTrigger`, `DialogCloseButton`, `DialogContent`, `DialogHeader`, `DialogFooter`, `DialogTitle`, `DialogDescription`
- Types: `DialogProps`, `DialogTriggerProps`, `DialogCloseButtonProps`, `DialogContentProps`, `DialogHeaderProps`, `DialogFooterProps`, `DialogTitleProps`, `DialogDescriptionProps`
```

Rules:

- List components first, then types on a separate line prefixed with `Types:`
- Use PascalCase for component/type names, camelCase for utilities and variants exports
- Order sub-components logically (parent → structural → interactive → descriptive)
- Include variant exports like `buttonVariants`, `alertVariants` when they exist

## 9. API Section

The **API** section is optional and should only appear for:

- HOCs with imperative APIs (e.g. `confirm()`)
- Components with complex prop surfaces that need structured documentation
- Modules with multiple exported functions

### 9.1 Prop Tables

Describe props in a list format, not a markdown table:

```mdx
### `DataTableProps<TData>`

Extends `div` props. Notable fields:

- `columns`, `data` — TanStack `ColumnDef` array and row data
- `loading`, `loadingRows` — skeleton instead of table
- `searchColumn` — column id for toolbar search (omit to hide search)
- `showColumnToggle` — column visibility menu (default `true`)
```

### 9.2 Imperative APIs

For functions like `confirm()`, document the signature, parameters, and return type:

```mdx
### `confirm(options: ConfirmDialogOptions): Promise<boolean>`

Resolves to `true` when the user confirms, `false` when cancelled or dismissed.

### `ConfirmDialogOptions`

- `title` — dialog title
- `description` — optional string or JSX
- `variant` — `'default'` | `'destructive'`
- `locale` — partial button label overrides
```

## 10. Installation Section

Include an **Installation** section only when the component requires a **peer dependency** that is not bundled with `shadcn-solid-components`:

```mdx
## Installation

This module expects `@tanstack/solid-table` (a peer dependency of the library).

```bash
pnpm add @tanstack/solid-table
```

```

Do **not** add an Installation section for components that have no external peer dependencies. The main library installation is covered at the project level, not per-component.

---

## 11. Cross-References and Links

- When mentioning another component from this library, use its display name and optionally link to its doc page: `[Dialog](/docs/components/dialog)`.
- When referencing an external library, link to its official documentation: `[Kobalte](https://kobalte.dev/docs/core/overview/dialog)`.
- Do not link to internal source files (e.g. `src/components/button/index.tsx`) from documentation.

---

## 12. Adding a New Doc Page Checklist

When adding documentation for a new component or HOC:

- [ ] Create `.mdx` file in the correct directory (`components/`, `hoc/`, or `lib/`)
- [ ] Frontmatter includes all required fields (`title`, `description`, `section`, `toc`)
- [ ] Opening paragraph explains what the component is and when to use it
- [ ] **Usage** section with the public import path
- [ ] **Exports** section listing all public symbols and types
- [ ] At least one **Examples** section with a `<ComponentPreview>`
- [ ] Example files in `docs/src/examples/` for complex demos
- [ ] **API** section if the component has complex props or imperative methods
- [ ] **Installation** section only if a peer dependency is required
- [ ] Verify the page appears in the docs navigation (auto-generated from frontmatter)
- [ ] Run `pnpm dev` in the `docs/` directory and verify the page renders correctly

---

## 13. Updating an Existing Doc Page

When modifying an existing doc page:

- If you add a new exported symbol, update the **Exports** section
- If you add a new variant or feature, add a corresponding **Example** subsection and example file
- If you change a prop name or type, update the **API** section
- If you deprecate a feature, add `status: "deprecated"` to the frontmatter and note the deprecation in the opening paragraph
- The `index.ts` file regenerates automatically — do not edit it manually

---

## 14. Common Mistakes to Avoid

- **Missing frontmatter** — the page will fail to render in the navigation
- **Wrong `section` value** — the page will appear in the wrong nav group
- **Relative import paths in code examples** — always use the `shadcn-solid-components/...` alias
- **Incomplete code examples** — every code block must be copy-paste runnable
- **Editing `index.ts`** — it is autogenerated and will be overwritten on the next build
- **Omitting the Exports section** — readers need to know every public symbol
- **Using `ComponentPreview` without the code block** — the preview must always include the source code
- **Hardcoded demo components inside `.mdx`** — extract complex demos to `examples/` files
